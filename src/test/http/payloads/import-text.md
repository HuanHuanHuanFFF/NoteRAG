# 2026-04-28 HashMap 与 ConcurrentHashMap 对比总结

来源：4.28 对话增量整理  
主题：HashMap、哈希冲突、开放寻址、红黑树、线程安全、ConcurrentHashMap

---

## 一、这块知识要解决什么问题

这块的核心不是背 API，而是理解：

> Map 要在“快速查找 key 对应 value”的同时，处理哈希冲突、扩容、并发访问这些问题。

可以分成两条主线：

1. **HashMap 如何在单线程下高效存取数据**
2. **ConcurrentHashMap 如何在多线程下保证安全且尽量不慢**

---

## 二、HashMap 的基本结构

HashMap 底层可以先理解为：

```text
数组 + 链表 / 红黑树
```

数组里的每个位置叫一个桶 bucket。

```text
table[0] -> null
table[1] -> Node -> Node -> Node
table[2] -> null
table[3] -> TreeNode
```

其中：

- 数组负责根据 hash 快速定位桶位
- 链表 / 红黑树负责处理同一个桶里的冲突元素

---

## 三、哈希冲突是什么

哈希冲突不是只指两个 key 的 hash 值完全一样。

更准确地说：

> 不同 key 经过 hash 和数组下标映射后，落到了同一个桶位。

比如：

```text
hash1 != hash2
但 index1 == index2
```

在 Java HashMap 里，下标计算大致是：

```java
index = hash & (table.length - 1);
```

所以冲突本质是：

> 多个 key 想放到同一个数组位置。

---

## 四、HashMap 如何解决冲突：拉链法

Java HashMap 使用的主要方案是 **拉链法**。

也就是：

```text
数组桶位 -> 链表 / 红黑树
```

冲突元素会挂到同一个桶后面。

例如：

```text
table[1] -> A -> B -> C
```

当桶内节点很少时，用链表更简单。

但如果所有 key 极端情况下都落到同一个桶，链表会退化：

```text
查找复杂度：O(n)
```

所以 Java 8 之后引入了红黑树兜底。

---

## 五、链表树化与退化

常见阈值：

```text
链表长度 >= 8：尝试树化
树节点数量 <= 6：退化回链表
数组容量 < 64：优先扩容，而不是树化
```

注意：

> 不是链表长度一到 8 就一定立刻树化。

如果数组容量还小于 64，HashMap 会认为：

> 当前冲突多，可能只是因为桶太少，应该先扩容让元素重新分散。

所以规则更准确是：

> 桶内链表过长时会尝试树化；但如果数组容量还比较小，会优先扩容。

---

## 六、为什么不是一开始就用红黑树

红黑树能把极端桶内查询从 O(n) 兜底到 O(log n)，但它也有成本：

- 节点结构更复杂
- 要维护颜色
- 插入 / 删除可能需要旋转和变色
- 小数据量下不一定比链表划算

所以 HashMap 的策略是：

> 节点少时用链表，结构简单；节点多到一定程度后转红黑树，避免极端退化。

---

## 七、红黑树最小理解

红黑树是：

> 一种近似平衡的二叉搜索树。

它不是 AVL 树。

AVL 更严格，强调左右子树高度差不超过 1。  
红黑树没这么严格，它是通过颜色规则保证树不会歪得太离谱。

红黑树完整规则可以记成：

```text
1. 节点不是红色就是黑色
2. 根节点是黑色
3. 空叶子 NIL 节点是黑色
4. 红节点不能连续，也就是红节点不能有红色孩子
5. 从任意节点到叶子 NIL 节点的黑色节点数相同，也就是黑高一致
```

最关键的是两条：

```text
红不相连
黑高一致
```

它们共同限制了最长路径不能比最短路径长太多，所以红黑树能保持 O(log n) 级别的查询、插入、删除。

红黑树维护平衡主要靠：

```text
变色 + 旋转
```

插入新节点通常先染成红色，因为这样不会直接破坏黑高一致；如果出现红红相连，再通过变色和旋转修复。

---

## 八、其他哈希冲突解决方案：开放寻址法

除了拉链法，还有一大类叫 **开放寻址法**。

核心思想是：

> 冲突后，不在桶后面挂链表，而是在数组里继续找下一个候选位置。

常见方式：

### 1. 线性探测

```text
index, index + 1, index + 2, ...
```

冲突了就一个个往后找。

### 2. 二次探测

```text
index + 1², index + 2², index + 3², ...
```

每次按平方步长跳。

### 3. 双重哈希

用第二个 hash 函数决定每次跳多远。

```text
step = hash2(key)
index, index + step, index + 2 * step, ...
```

开放寻址法没有链表兜底，所以更怕数组太满。

---

## 九、开放寻址法的扩容和删除问题

开放寻址法通常也有扩容机制，而且非常依赖负载因子控制。

因为数组越满，冲突后探测路径越长。

普通开放寻址法通常会把负载因子控制在较低范围，例如大致 0.5 到 0.75 左右。不同语言和实现会不一样。

扩容时，本质上需要把元素重新分布到新数组里。

更准确地说：

> 不一定重新计算 key 的 hash 值，但容量变了，桶位和探测路径都会变，所以必须重新定位。

开放寻址法还有一个额外麻烦：删除。

如果简单把元素位置清空，可能会破坏后续元素的探测链。  
所以很多实现会使用 tombstone 墓碑标记。

---

## 十、Java HashMap 的负载因子和容量

Java HashMap 默认：

```text
默认初始容量：16
默认负载因子：0.75
```

扩容条件大致是：

```text
size > capacity * loadFactor
```

例如容量为 16：

```text
16 * 0.75 = 12
```

元素数量超过阈值后，会触发扩容。

---

## 十一、为什么 HashMap 容量必须是 2 的幂

HashMap 算下标时使用：

```java
index = hash & (n - 1);
```

只有当 n 是 2 的幂时，这个操作才等价于：

```java
index = hash % n;
```

如果 n 不是 2 的幂，还用 `hash & (n - 1)`，会导致很多桶永远用不上。

例如 n = 10：

```text
n     = 10 = 1010
n - 1 = 9  = 1001
```

`hash & 9` 只能产生部分下标，桶分布会非常不均。

所以 HashMap 会保证容量是 2 的幂。

---

## 十二、初始化 capacity 会怎么处理

如果传入的初始化容量不是 2 的幂，HashMap 会把它处理成：

> 大于等于传入值的最小 2 的幂。

例如：

```text
10  -> 16
17  -> 32
33  -> 64
100 -> 128
```

注意：

> initialCapacity 不是“最多能放多少个元素”，而是桶数组容量的参考值。

如果预计放 expectedSize 个元素，为了避免扩容，可以先估算：

```java
(int)(expectedSize / 0.75f) + 1
```

然后 HashMap 内部还会再向上取 2 的幂。

例如 expectedSize = 100：

```text
100 / 0.75 ≈ 134
134 -> 256
```

最终容量大概率是 256。

---

## 十三、put 流程最小模型

`map.put(key, value)` 大致流程：

```text
1. 计算 key 的 hash
2. 根据 hash 和 table 长度计算桶下标
3. 如果桶为空，直接放入新节点
4. 如果桶不为空，说明冲突
   - key 已存在：覆盖 value
   - key 不存在：插入链表或红黑树
5. 插入后检查是否需要树化
6. 检查 size 是否超过阈值，必要时扩容
```

可以压成一条线：

```text
put
 ↓
算 hash
 ↓
算桶下标
 ↓
桶空：直接放
桶不空：比较 key
 ↓
相同 key：覆盖 value
不同 key：链表 / 红黑树插入
 ↓
必要时树化 / 扩容
```

---

## 十四、HashMap 为什么不是线程安全的

HashMap 不适合多线程并发写。

原因是 `put/remove/resize` 这些操作内部不是原子的。

一次 `put` 看起来是一行代码：

```java
map.put(k, v);
```

但内部可能包含多步：

```text
算 hash
算桶下标
看桶里有没有节点
插入 / 覆盖
size++
判断是否扩容
```

多个线程交叉执行这些步骤，就可能出问题。

常见问题：

### 1. 数据覆盖 / 丢失

两个线程同时往同一个桶插入，可能互相覆盖。

### 2. size 不准确

`size++` 不是原子操作。

它大致是：

```text
读取 size
加 1
写回 size
```

两个线程同时执行，可能明明加了两次，最后只加了一次。

### 3. 扩容时结构异常

并发 resize 时，内部结构可能不一致。

老版本 JDK 中，HashMap 并发扩容甚至可能导致链表成环。新版本实现有所改进，但 HashMap 仍然不保证并发写安全。

---

## 十五、多线程只读可以吗

要分情况：

### 初始化完成后只读

通常可以。

### 一边读一边写

不安全。

一个线程 `get`，另一个线程同时 `put/remove/resize`，读线程可能看到中间状态。

更稳的说法：

> HashMap 可以用于多线程只读，但不能用于并发读写或并发写。

---

## 十六、并发场景下可以用什么

常见选择：

### 1. ConcurrentHashMap

最常用，专门为并发场景设计。

### 2. Collections.synchronizedMap

给 HashMap 外面包一层同步锁。

```java
Map<K, V> map = Collections.synchronizedMap(new HashMap<>());
```

简单，但并发性能一般。

### 3. Hashtable

老式线程安全 Map。基本不推荐新代码优先使用。

---

## 十七、Hashtable / synchronizedMap / ConcurrentHashMap 的差异

### Hashtable / synchronizedMap

思路更接近：

> 整张表一把大锁。

优点：简单，安全。  
缺点：锁粒度粗，并发性能差。

如果线程 A 操作 key1，线程 B 操作 key2，即使它们落在不同桶里，也可能要排队。

### ConcurrentHashMap

思路是：

> 读尽量无锁，写只控制必要的局部范围。

它不是整张表一把大锁，而是尽量缩小同步范围。

---

## 十八、ConcurrentHashMap 为什么存在

可以用一句话概括：

> HashMap 并发不安全；Hashtable 整体加锁太粗；ConcurrentHashMap 用更细粒度的并发控制，在保证线程安全的同时提高并发性能。

可以理解为三层递进：

```text
HashMap：单线程好用，多线程写不安全
Hashtable / synchronizedMap：线程安全，但整表加锁，性能差
ConcurrentHashMap：线程安全，同时尽量减少不必要的锁竞争
```

---

## 十九、ConcurrentHashMap 的核心思路

Java 8 之后，可以先用这个简化模型理解：

### get

通常不加锁。

因为读操作只沿着桶结构查找，不修改结构。

### put

分情况：

```text
桶为空：尝试用 CAS 放入节点
桶不为空：锁住当前桶的头节点，修改这个桶内的链表或树
```

这背后的原则是：

> 能不用锁就不用锁；必须同步时，只同步局部。

---

## 二十、CAS 是什么

CAS 全称：

```text
Compare And Swap
```

中文常说：比较并交换。

它的逻辑是：

```text
如果当前位置仍然是我预期的旧值
就把它改成新值
否则说明别人已经改过了，我失败 / 重试
```

在 ConcurrentHashMap 里，桶为空时，如果多个线程都想抢这个桶位：

```text
table[i] == null
```

会尝试 CAS：

```text
如果 table[i] 仍然是 null
就把新节点放进去
否则失败重试
```

所以空桶插入不需要先加锁。

---

## 二十一、synchronized 是什么

`synchronized` 是 Java 里的同步机制。

基本语法：

```java
synchronized (lock) {
    // 需要互斥执行的代码
}
```

意思是：

> 进入代码块前，线程必须先拿到 lock 这个对象关联的 monitor 锁；执行完后释放锁。

例子：

```java
private final Object lock = new Object();

public void add() {
    synchronized (lock) {
        count++;
    }
}
```

这里的 `lock` 只是作为一把锁。  
它不一定有业务意义。

常见写法用 `private final Object lock = new Object();` 是因为：

- `private`：避免外部代码拿到这把锁乱用
- `final`：避免锁对象被换掉

---

## 二十二、锁住对象是什么意思

`synchronized(lock)` 不是把对象物理锁住，让别人完全不能看。

更准确地说：

> 所有也使用同一把锁 `lock` 的 synchronized 代码块，必须互斥执行。

如果另一个线程没有进入 `synchronized(lock)`，锁本身不会自动拦住它访问字段。

所以锁是一种约定：

> 所有访问共享数据的相关代码，都要使用同一把锁，才能保证互斥和可见性。

方法上的 `synchronized`：

```java
public synchronized void add() {
    count++;
}
```

等价于锁当前对象：

```java
synchronized (this) {
    count++;
}
```

静态 synchronized 方法锁的是类对象：

```java
synchronized (MyClass.class) {
    // ...
}
```

---

## 二十三、ConcurrentHashMap 具体锁哪里

桶为空时：

```text
CAS 抢桶位
```

桶不为空时：

```text
锁住当前 key 命中的桶头节点
```

可以粗略理解为：

```java
synchronized (桶头节点) {
    修改这个桶里的链表 / 树
}
```

注意：

> 锁的是当前桶，不是整张表，也不是旁边的桶。

所以：

```text
线程 A 修改 table[3]
线程 B 修改 table[99]
```

它们可以并发。

但如果两个线程都修改同一个桶：

```text
线程 A 修改 table[3]
线程 B 也修改 table[3]
```

就需要同步。

---

## 二十四、ConcurrentHashMap 的读一致性边界

ConcurrentHashMap 不是数据库，没有 MVCC 那种事务快照读。

它保证的是：

> 单个操作线程安全；单个 key 的已完成更新对后续相关读取有可见性保证。

但它不保证：

> 遍历、size、多 key 组合读取一定看到整张 Map 某个瞬间的一致快照。

所以：

```java
map.get(k)
```

通常可以不加锁，并且读到安全的结果。

但：

```java
map.size()
map.forEach(...)
```

在并发更新时可能只是弱一致视图。

---

## 二十五、弱一致是什么意思

弱一致不是乱读。

更准确：

> 可以读到旧状态，也可能看到部分更新后的状态，但不会读到被破坏的内部结构。

例如链表桶原来是：

```text
A -> B -> null
```

另一个线程正在插入 C：

```text
A -> B -> C -> null
```

读线程可能看到：

```text
A -> B -> null
```

也可能看到：

```text
A -> B -> C -> null
```

这两种都可以。

但不能看到结构损坏，例如乱指、成环、非法断裂。

---

## 二十六、链表修改时为什么还能遍历

链表结构相对简单，读线程可以沿着已经安全发布的 `next` 指针往后走。

写线程可能正在追加新节点，读线程不一定能看到最新节点，但它能看到一个安全结构。

所以：

> 链表桶的读可以弱一致地遍历。

这不是数据库快照读，也不是强一致读。

---

## 二十七、红黑树桶为什么更复杂

红黑树写入或删除时可能发生：

- 左旋
- 右旋
- 根节点变化
- 父子关系变化
- 颜色变化

如果读线程正在按树结构查找，而写线程同时旋转，就可能影响读路径。

所以 ConcurrentHashMap 的树化桶有额外控制结构，常见理解为 `TreeBin`。

这里不用展开源码，只记：

> 链表桶读路径相对简单；红黑树桶因为旋转会改变树结构，所以需要额外机制保证读写安全。

---

## 二十八、ConcurrentHashMap 不保证复合操作自动原子

例如：

```java
if (!map.containsKey(k)) {
    map.put(k, v);
}
```

这不是原子的。

两个线程可能同时判断不存在，然后都 put。

应使用：

```java
map.putIfAbsent(k, v);
```

所以要记住：

> ConcurrentHashMap 保证单个方法的线程安全，但多个方法组合起来不一定天然原子。

---

## 二十九、HashMap vs ConcurrentHashMap 最终对比

| 维度 | HashMap | ConcurrentHashMap |
|---|---|---|
| 线程安全 | 不安全 | 安全 |
| 典型场景 | 单线程 / 多线程只读 | 多线程并发读写 |
| 冲突处理 | 数组 + 链表 / 红黑树 | 数组 + 链表 / 红黑树 |
| 写入同步 | 无同步控制 | CAS + 局部 synchronized |
| 读操作 | 无并发保证 | 通常无锁读，弱一致 |
| 扩容并发 | 不安全 | 有并发控制 |
| 适用性 | 普通 Map | 并发 Map |

---

## 三十、面试复述版

可以这样回答：

> HashMap 底层是数组加链表或红黑树。put 时先计算 hash，再通过 `hash & (n - 1)` 定位桶位；如果发生冲突，就在桶内用链表或红黑树处理。链表过长时会尝试树化，阈值通常是 8，但如果数组容量小于 64 会优先扩容；树节点数量降到 6 左右会退化回链表。HashMap 默认负载因子是 0.75，容量会保持为 2 的幂，这样可以用位运算计算下标，并优化扩容迁移。

> HashMap 不是线程安全的，因为 put、remove、resize 这些操作不是原子的，并发写可能导致数据覆盖、size 不准或结构异常。如果需要并发读写，一般使用 ConcurrentHashMap。

> ConcurrentHashMap 不是简单整张表加锁，而是用更细粒度的并发控制。get 通常不加锁；put 时如果桶为空，会用 CAS 原子放入节点；如果桶不为空，会锁住当前桶的头节点，在这个桶内修改链表或树。这样不同桶上的写操作可以并发，比 Hashtable 或 synchronizedMap 的整表加锁更高效。不过 ConcurrentHashMap 不是数据库，不提供整张 Map 的一致性快照；遍历和 size 在并发修改时是弱一致的，并且多个方法组合起来也不一定天然原子，需要使用 putIfAbsent 这类复合原子方法。

## 三十一、最小记忆版

```text
HashMap：
  单线程 Map
  数组 + 链表 / 红黑树
  默认负载因子 0.75
  容量保持 2 的幂
  并发写不安全

ConcurrentHashMap：
  并发 Map
  读尽量无锁
  空桶 CAS
  非空桶锁当前桶
  弱一致读，不是数据库快照
  单方法安全，组合操作不一定原子
```
